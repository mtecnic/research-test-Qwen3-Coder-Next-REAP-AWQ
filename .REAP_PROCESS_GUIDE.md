## Overview

The REAP pipeline takes an MoE (Mixture-of-Experts) model and:

1. **Observes** which experts matter by running calibration data through the model
2. **Merges** observations from multiple diverse datasets
3. **Prunes** the least important experts (structural removal, not zeroing)
4. **Quantizes** the pruned model to W4A16 using AWQ
5. **Evaluates** the final model on code, reasoning, tool-use, and chat benchmarks

The result is a smaller, faster model that retains the routing structure and quality of the original.

## Hardware Requirements

### VRAM Planning

The model needs to fit in VRAM for reasonable speed. If it doesn't fit, it offloads to CPU RAM and becomes ~30-75x slower.

| Model Size (BF16) | 4x 3090 (96GB) | 2x A100 (160GB) | 8x A100 (640GB) |
|-------------------|-----------------|------------------|-------------------|
| < 80GB | Full VRAM, fast | Full VRAM, fast | Full VRAM, fast |
| 80-150GB | **CPU offload, very slow** | Full VRAM, fast | Full VRAM, fast |
| 150GB+ | CPU offload, very slow | CPU offload, slow | Full VRAM, fast |

**Our experience with 149GB model on 96GB VRAM:**
- Phase 1 observation: ~2.4 min/sample (vs ~3 sec if it fit)
- 256 samples × 4 datasets = **~40 hours** for observation alone
- Total pipeline: **~3 days**

**If the model fits in VRAM, the entire pipeline takes under 4 hours.**

### Disk Planning

**CRITICAL: Check disk space before starting.** Our AWQ quantization completed 15 hours of calibration then failed on save because the disk was full.

You need space for:
- Original BF16 model (keep it — re-pruning from scratch takes days)
- Pruned BF16 model (~80% of original size)
- AWQ quantized model (~25-30% of original size)
- Observation files (~100MB per dataset, negligible)

**Example for Qwen3-Coder-Next:**
- Original: 149GB
- Pruned (20%): 120GB
- AWQ output: ~35GB
- **Total needed: ~305GB**

Check before starting:
```bash
df -h /data
```

### CPU RAM

- Observation: model overflow goes to CPU RAM (need model_size - VRAM_size + ~10GB headroom)
- AWQ: explicitly allows 100GB CPU overflow via `max_memory`
- Our setup: 128GB system RAM was sufficient

## Lessons Learned the Hard Way

These cost us hours or days. Don't repeat them.

### 1. Never load AWQ models via HuggingFace for observation
The `compressed-tensors` format decompresses ALL weights back to BF16 on load (~85 minutes for a 512-expert model). This defeats the entire purpose. Always use the BF16 model for observation/pruning. AWQ models are only fast when served via vLLM.

### 2. Always check disk space before long-running saves
AWQ calibration takes 15+ hours. If the disk fills during `save_pretrained`, you lose all that work and have to restart from scratch. There's no resume/checkpoint.

### 3. Never delete the pruned BF16 model to free space
It takes 2+ days to regenerate (observation + pruning). Find something else to delete. The AWQ output is what you want to keep long-term, but the pruned BF16 is your insurance policy until AWQ is verified.

### 4. Each pipeline phase MUST be a separate process
The BF16 model + observer state + PyTorch CUDA allocations can exceed memory if phases overlap. Separate processes guarantee clean GPU state — when a process exits, ALL GPU memory is freed by the driver. This is not optional.

### 5. Set CUDA_HOME before Phase 3
The pruning phase triggers deepspeed compilation which needs `CUDA_HOME`. Without it, you get `MissingCUDAException` after waiting hours for observation to complete.

```bash
export CUDA_HOME=/data/cuda-12.8  # or wherever your CUDA toolkit is
```

### 6. Previous pruning at 40% was too aggressive
For Qwen3-Coder-Next with 512 experts, removing 40% (205 experts) caused quality degradation. 20% (102 experts) is the sweet spot we settled on. Start conservative — you can always prune more, but you can't un-prune.

### 7. Super-expert preservation matters
Some experts rarely fire but are critical when they do. The `--perserve_super_experts true` flag protects these. Without it, pruning removes them based on low average REAP scores, causing failures on specific inputs.

### 8. Diverse calibration rescues experts
Calibrating only on code data marks some experts as unimportant that are actually critical for reasoning or general language. Using 4 diverse datasets (code, web text, creative writing, math) gives balanced pruning decisions.

### 9. vLLM tool parser must match the model
Using `--tool-call-parser hermes` for a Qwen3-Coder model causes tool-calling tests to fail. The model generates tools in its own XML format. Use `--tool-call-parser qwen3_coder` with the custom parser installed. Check the model directory for custom parsers.

### 10. AWQ ignore list is model-specific
Certain layers are fragile to quantization. For Qwen3-Coder-Next:
- MoE routers (`mlp.gate`) — routing precision is critical
- Shared expert gates (`mlp.shared_expert_gate`)
- DeltaNet linear attention (`linear_attn.conv1d`, `in_proj_a/b`)
- Output head (`lm_head`)

Get this wrong and the quantized model produces garbage. Check the model architecture before writing the AWQ recipe.

## Pipeline Phases

### Phase 1: Observation

**What it does:** Runs calibration samples through the model and records per-expert metrics using forward hooks.

**Key metrics collected:**
- `expert_frequency` — how often each expert is routed to
- `ean_sum` — total L2 norm of expert output activations
- `reap` — activation norm weighted by router probability (primary pruning metric)
- `max_activations` — peak activation (identifies super-experts)

**How hooks work:** Attached to each `SparseMoeBlock`, they intercept the router output and expert activations. Statistics accumulate on **CPU memory** (not GPU) to avoid OOM. After all samples, saved as a `.pt` file.

**Key parameters:**
- `--samples_per_category` — number of calibration samples (we used 256)
- `--model_max_length` — max tokens per sample (we used 512)
- `--record_pruning_metrics_only true` — skip merging metrics to save memory/time

**Each dataset runs as a separate process.** Model loads fresh each time (~100s for BF16 from safetensors). This wastes time on model loading but guarantees clean GPU state.

### Phase 2: Merge

**What it does:** CPU-only script that loads all observation `.pt` files and combines them.

- Additive metrics (frequency, EAN sum) are summed
- Derived metrics (REAP, EAN mean) are recomputed from merged sums
- Max activations take element-wise max across datasets

**Output:** `artifacts/<model>/combined/all/observations_<N>_cosine.pt`

### Phase 3: Prune

**What it does:** Loads the BF16 model, loads the merged observations, removes the bottom N% of experts per layer.

**Process:**
1. Rank experts per layer by REAP score (low = least important)
2. Optionally protect super-experts from pruning
3. Remove expert modules from `nn.ModuleList`
4. Remove corresponding rows from router weight matrix
5. Update `config.num_experts`
6. Collect state dict (handles CPU-offloaded parameters)
7. Save with `safe_serialization=True`

**The save is complex** because the model is partially on CPU (offloaded). The code at `prune.py:213-331` rebuilds the state dict by:
- Collecting GPU-resident params directly
- Fetching CPU-offloaded params from `_hf_hook.weights_map`
- Re-indexing expert numbering (expert 500 → expert 380 after pruning)

### Phase 4: AWQ Quantization

**What it does:** Quantizes the pruned model to W4A16 using activation-aware weight quantization.

**Memory management:**
- `max_memory = {i: "20GiB" for i in range(n_gpus)}` — caps GPU usage
- `max_memory["cpu"] = "100GiB"` — allows CPU overflow for grid search
- `model.to("cpu")` before save — frees GPU for serialization

**AWQ recipe:**
- W4A16, symmetric, group_size=32
- Ignore list: routers, shared expert gates, DeltaNet internals, lm_head
- 12 grid points, 256 calibration samples
- Sequential targets by decoder layer class

### Phase 5: Evaluation

**What it does:** Serves the AWQ model via vLLM and runs benchmarks.

**vLLM serve args that matter:**
- `--enable-expert-parallel` — required for MoE models
- `--tool-call-parser qwen3_coder` — must match model's tool format
- `--max-num-seqs 16` — keep batch small for MoE memory
- `--gpu-memory-utilization 0.93` — leave some headroom

**Custom eval script** (`~/model_ab_eval.py`) tests:
- Code (4 tests): merge sorted lists, debug binary search, explain regex, DP palindrome
- Reasoning (4 tests): arithmetic, work rates, combinatorics, logic puzzles
- Scaffold (4 tests): tool calling, multi-tool planning, structured JSON, multi-turn tool use
- Chat (5 tests): summarization, Q&A, instruction following, creative writing, multi-turn

## Running the Pipeline

### Quick Start

```bash
# 1. Check prerequisites
df -h /data                           # Enough disk?
nvidia-smi                            # GPUs free?
echo $CUDA_HOME                       # Set?

# 2. Run the full pipeline
cd /data/reap
nohup bash reap_pipeline.sh > /tmp/reap_pipeline.log 2>&1 &
disown

# 3. Monitor
bash /tmp/check_reap.sh               # If the check script exists
# Or manually:
tail -c 500 /tmp/reap_pipeline.log | tr '\r' '\n' | tail -5
```

### Running Phases Individually

If you need to restart from a specific phase:

**Phase 1 only (one dataset):**
```bash
cd /data/reap && source .venv/bin/activate
export CUDA_HOME=/data/cuda-12.8

CUDA_VISIBLE_DEVICES=0,1,2,3 python3 src/reap/prune.py \
    --model-name /data/models/YOUR_MODEL \
    --dataset-name theblackcat102/evol-codealpaca-v1 \
    --compression-ratio 0.2 \
    --prune-method reap \
    --profile false \
    --do-eval false \
    --run_observer_only true \
    --distance_measure cosine \
    --seed 42 \
    --output_file_name observations_256_cosine.pt \
    --samples_per_category 256 \
    --model_max_length 512 \
    --record_pruning_metrics_only true \
    --smoke_test false
```

**Phase 3 only (prune from existing observations):**
```bash
CUDA_VISIBLE_DEVICES=0,1,2,3 python3 src/reap/prune.py \
    --model-name /data/models/YOUR_MODEL \
    --dataset-name combined \
    --compression-ratio 0.2 \
    --prune-method reap \
    --profile false \
    --do-eval false \
    --distance_measure cosine \
    --seed 42 \
    --output_file_name observations_256_cosine.pt \
    --perserve_super_experts true \
    --perserve_outliers false \
    --smoke_test true
```

**Phase 4 only (AWQ from pruned model):**
```bash
CUDA_VISIBLE_DEVICES=0,1,2,3 python3 quantize_awq.py \
    --model-path artifacts/YOUR_MODEL/combined/pruned_models/reap-perserve_super-seed_42-0.20 \
    --save-dir /data/models/YOUR_MODEL-REAP-AWQ
```

## Monitoring

### Create a status check script

```bash
cat > /tmp/check_reap.sh << 'EOF'
#!/bin/bash
echo "=== REAP Pipeline Status ==="
if ps aux | grep "reap_pipeline" | grep -v grep > /dev/null; then
    echo "STATUS: RUNNING"
else
    echo "STATUS: NOT RUNNING"
fi
grep -E "\[Phase" /tmp/reap_pipeline.log 2>/dev/null | tail -1
tail -c 500 /tmp/reap_pipeline.log 2>/dev/null | tr '\r' '\n' | grep -v "^$" | tail -3
PID=$(ps aux | grep "prune.py\|quantize" | grep -v grep | awk '{print $2}' | head -1)
if [ -n "$PID" ]; then
    ELAPSED=$(ps -o etimes= -p $PID 2>/dev/null)
    echo "Runtime: $((ELAPSED/3600))h $((ELAPSED%3600/60))m"
fi
nvidia-smi --query-gpu=index,memory.used --format=csv,noheader
echo "==========================="
EOF
chmod +x /tmp/check_reap.sh
```

### Key things to watch

- **GPU memory should stay flat** during observation — if it's climbing, hooks are leaking to GPU instead of CPU
- **CPU memory grows slowly** during observation — this is normal (accumulating statistics)
- **Disk usage** — check before Phase 3 save and Phase 4 save
- **Temperature** — passive-cooled GPUs should stay under 80C

## Tuning for Different Models

### Compression ratio

- **Start at 10-20%** for a first run. Check eval scores.
- **Previous experience:** 40% on Qwen3-Coder-Next was too much. 20% retained quality.
- MoE models with fewer experts (e.g., 8-16) are more sensitive — try 10% first.
- MoE models with many experts (256-512) can tolerate 20-30%.

### Samples and sequence length

- **256 samples at 512 tokens** is a reasonable minimum for good REAP estimates
- More samples help but with diminishing returns — expert routing patterns stabilize quickly
- If the model fits in VRAM, use 1024 samples at 2048 tokens (costs minutes, not days)
- If CPU offloading, keep it short — every sample is 2+ minutes

### Pruning method

Available methods in order of recommendation:
1. `reap` — best overall (frequency × magnitude, normalized)
2. `weighted_ean_sum` — good alternative (EAN × router probability)
3. `frequency` — simple baseline, surprisingly effective
4. `ean_sum` / `ean_mean` — magnitude only, misses rarely-used-but-important experts

### Calibration datasets

Use at least 2-3 diverse datasets. Match to the model's intended use:
- **Code model:** code + general text + math
- **General model:** web text + instruction following + reasoning
- **Multilingual:** include samples from target languages

Available in REAP's `DATASET_REGISTRY`:
- `theblackcat102/evol-codealpaca-v1` — code
- `allenai/c4` — web text
- `euclaise/WritingPrompts_curated` — creative writing
- `allenai/tulu-3-sft-personas-math` — math
- `ise-uiuc/Magicoder-Evol-Instruct-110K` — code (alternative)
- `cais/mmlu` — academic knowledge

### AWQ ignore list

**Always ignore:**
- MoE router weights (`mlp.gate`) — routing must stay precise
- `lm_head` — output head

**Model-specific ignores:**
- Shared expert gates (if the architecture has them)
- Non-standard attention mechanisms (DeltaNet, linear attention, Mamba)
- Any layer the model README says is quantization-sensitive

## Troubleshooting

### "CUDA_HOME does not exist"
```bash
export CUDA_HOME=/path/to/cuda  # e.g., /data/cuda-12.8 or /usr/local/cuda
```

### "No space left on device" during save
Check `df -h`. Free space by removing models you don't need. **Never delete the pruned BF16 model** — it's not reproducible without re-running the full observation pipeline.

### AWQ model decompresses on load (takes 85+ minutes)
You're loading a `compressed-tensors` model via HuggingFace. This is expected. Use vLLM to serve it instead — vLLM handles quantized inference natively without decompression.

### Observation is extremely slow (~3 min/sample)
Model is CPU-offloaded. This is normal when the model doesn't fit in VRAM. Options:
- Reduce samples/seq length
- Use hardware with more VRAM
- Accept the long runtime and use `nohup`

### Smoke test fails after pruning
Shape mismatch in router weights means the pruning didn't properly update the router. Check that `model_util.py` has the correct attribute names for your model architecture (especially `router`, `experts`, `num_experts`).

### vLLM OOM during CUDA graph capture
Add `--enforce-eager` to skip CUDA graphs, or reduce `--gpu-memory-utilization`. Also try `--enable-expert-parallel` which distributes experts across GPUs differently.

### Tool calling tests fail in eval
Wrong tool parser. Check what parser the model needs:
- Qwen3-Coder: `qwen3_coder`
- Llama: `llama`
- Hermes format: `hermes`
